Keystone API Flow Diagram

프론트엔드 기준 API 흐름 도식화 · 로그인 / 일정 / 장소 / 틈새 추천 / 날씨 알림

1. 로그인 / 토큰 흐름

LoginScreen
이메일/비밀번호 입력 또는 소셜 로그인 시작
→
Login API
access_token / refresh_token 수신
POST /api/auth/login
→
AsyncStorage 저장
access_token refresh_token user_id nickname
→
App Bootstrap
토큰 있으면 Main 진입 없으면 Login/Onboarding

2. API Client / 토큰 재발급 흐름

apiClient
모든 API 요청에 Authorization Bearer 추가
→
401 / 403 발생
access_token 만료 가능성
→
Refresh API
refresh_token body 포함
POST /api/auth/refresh
→
토큰 갱신
새 access_token 저장 기존 요청 재시도
→
실패 시
토큰 삭제하지 않음 로그인 상태 유지 서버 장애 fallback

3. 메인 일정 조회 흐름

MainScreen
앱 진입 / focus 시 서버 일정 목록 조회
→
Trip List
timeout 5초 권장 로컬 fallback 사용 여부 정책 필요
GET /api/trips?status=ALL
→
Trip Detail Hydrate
메인 카드 상세 보강 요청 수 제한 필요
GET /api/trips/{tripId}
→
Main Card
진행중/예정 일정 카드 표시 첫 장소명 / 날짜 / 지역 표시

4. Plan.A 저장 / 장소 시간 반영 흐름

PlanAScreen
장소 추가 시간 수정 메모 수정
→
Local Save
PlanAStorage 저장 draft cache 보존
→
Trip Update
PUT /api/trips/{tripId}
→
Existing Place PATCH
기존 장소 시간/메모 수정 병렬 대신 순차 처리 권장
PATCH /api/plans/{tripPlaceId}
→
New Place Add
신규 장소 서버 추가
POST /api/trips/{tripId}/days/{day}/locations

5. 장소 검색 / 상세 분석 흐름

AddScheduleLocation
키워드 검색 장소 선택 상세 보기
→
Place Search
GET /api/places/search?query={keyword}
→
Place Detail
GET /api/places/{googlePlaceId}
→
AI Summary
GET /api/places/{googlePlaceId}/summary
→
Polling 주의
기존 최대 20요청 가능 MVP에서는 2회 이하 권장

6. 틈새 추천 / SSE 흐름

OngoingSchedule
현재 Day의 장소 간 빈 시간 계산
→
Gap List
GET /api/trips/{tripId}/gaps
→
Gap Recommend Stream
progress / place / done 이벤트 수신
POST /api/trips/{tripId}/gaps/recommend/stream
→
GapRecommendationCard
추천 장소 5개 표시 선택 시 Plan.A 이동
→
Plan.A 반영
선택한 trip/day에만 추가 gapSelectedPlace.day 필수

7. 날씨 알림 흐름

Weather Scheduler
서버에서 악천후 조건 감지 테스트 트리거와 분리 필요
→
Notification List
GET /api/notifications/{userId}
→
MainScreen
날씨 알림 카드 표시 대안 추천 진입
→
Replace / Dismiss
POST /api/notifications/{id}/replace/{newPlaceId}
POST /api/notifications/{id}/dismiss

8. 현재 장애 의심 지점

프론트 요청 과다 가능 지점
  • Main 진입 시 getTrips + getTripDetail 연쇄 호출
  • Ongoing focus마다 getTripDetail 재조회
  • 장소 상세 polling에서 API 다중 호출
  • Plan.A 저장 시 PATCH / ADD 요청 다수 발생
  • SSE 스트림 중복 실행 가능성
백엔드 확인 필요 지점
  • DB connection pool exhaustion
  • slow query / lock wait
  • SSE 처리 중 DB connection 장시간 점유
  • /api/auth/refresh 403 HTML 응답
  • Bearer 토큰 포함 요청에서만 500/502 발생
MVP 안정화 요청 수 축소 timeout 단축 fallback 정책 분리 SSE 중복 방지
